안녕하세요, 구독자 여러분! 벌써 한 해의 절반이 지나가고 있네요. 다들 상반기 프로젝트 마무리는 잘하고 계시는지 모르겠습니다. 

어제 퇴근길에 해외 테크 뉴스를 훑어보다가 흥미로운 소식을 하나 발견했습니다. 글로벌 뷰티 리테일 대기업인 얼타(Ulta)에서 다가오는 2026년 7월을 겨냥해 최대 50% 할인 혜택을 제공하는 프로모션 코드(Promo Code) 캠페인을 대대적으로 준비하고 있다는 내용이었는데요. 뷰티 테크(Beauty Tech) 기기부터 메이크업 제품까지 대폭 할인한다는 소식이었습니다.

일반 소비자분들이라면 "와, 대박! 7월에 직구 장바구니 좀 채워야겠네!" 하겠지만, 15년 차 백엔드 엔지니어인 제 머릿속에는 본능적으로 다른 생각이 스쳐 지나갔습니다.

> "와... 저거 담당 개발 팀원들, 트래픽 폭탄 맞을 준비 하느라 벌써부터 야근하고 있겠구나..." 

맞습니다. 대규모 프로모션 코드 이벤트는 마케팅 팀에게는 축제지만, 플랫폼을 지탱하는 엔지니어들에게는 **'합법적인 디도스(DDoS) 공격'**이 시작되는 날이나 다름없거든요. 

## 작년 가을, 저를 밤샘하게 만들었던 '선착순 쿠폰'의 악몽

이 뉴스를 보니 문득 작년 가을, 저희 팀이 진행했던 대규모 이커머스 개편 프로젝트가 떠올랐습니다. 당시 마케팅 부서에서 "선착순 1만 명에게 50% 할인 쿠폰을 뿌리는 이벤트를 하겠다"고 선언했었죠. 

당시 기존의 레거시 쿠폰 시스템은 동시성(Concurrency) 처리가 전혀 고려되어 있지 않은 상태였습니다. 트래픽이 몰리면 데이터베이스에 그대로 락(Lock)이 걸려 전체 결제 시스템이 마비되기 일쑤였죠. 

> "형님, 이번 이벤트 때 서버 터지면 우리 서비스 결제 전체가 먹통이 됩니다. 어떻게 하죠?"

식은땀을 흘리며 저를 바라보던 주니어 팀원의 눈빛이 아직도 생생합니다. 결국 저희 팀은 기존의 RDBMS 중심의 쿠폰 검증 로직을 전면 폐기하고, **Redis를 활용한 분산 락(Distributed Lock)**과 **메시지 큐(Message Queue)** 기반의 비동기 아키텍처로 긴급 전환을 결정했습니다. 

## 동시성 제어: 찰나의 순간에 서버를 지켜내는 기술

쿠폰 적용이나 프로모션 코드 발급에서 가장 중요한 것은 **"동일한 자원에 대한 동시 접근을 어떻게 안전하게 제어할 것인가"**입니다. 50% 할인이라는 파격적인 조건이 걸리면 수만 명의 사용자가 동시에 '적용' 버튼을 누릅니다. 이때 제대로 제어하지 않으면 준비된 수량보다 더 많은 쿠폰이 발급되는 '초과 발급(Over-issue)' 버그가 발생하거나, DB Deadlock으로 인해 시스템이 완전히 다운될 수 있습니다.

당시 저희 팀이 구축했던 아키텍처의 핵심 컨셉을 코드로 간단히 재구성해 보았습니다. Redis를 활용해 분산 환경에서도 안전하게 동시성을 제어하고 멱등성(Idempotency)을 보장하는 로직입니다.

```typescript
// Redis를 활용한 분산 락(Distributed Lock) 기반 프로모션 코드 적용 예시
async function applyPromoCode(userId: string, promoCode: string): Promise<boolean> {
    const lockKey = `lock:promo:${promoCode}:${userId}`;
    const acquireTimeout = 3000; // 3초 동안 락 획득 시도
    
    // 1. Redis 분산 락 획득 시도 (동일 사용자의 중복 요청 및 어뷰징 방지)
    const isLocked = await redis.set(lockKey, "processing", "NX", "PX", acquireTimeout);
    
    if (!isLocked) {
        throw new Error("이미 처리 중인 요청입니다. 잠시만 기다려주세요.");
    }
    
    try {
        // 2. 프로모션 코드의 유효성 및 잔여 수량 확인 (Redis In-Memory 조회로 DB 부하 최소화)
        const promoStockKey = `promo:stock:${promoCode}`;
        const currentStock = await redis.get(promoStockKey);
        
        if (currentStock !== null && parseInt(currentStock) <= 0) {
            throw new Error("준비된 할인 혜택이 모두 소진되었습니다.");
        }
        
        // 3. 멱등성 검증 (이미 이 코드를 적용해 혜택을 받았는지 확인)
        const hasUsed = await db.userPromoHistory.exists({ userId, promoCode });
        if (hasUsed) {
            throw new Error("이미 사용하신 프로모션 코드입니다.");
        }
        
        // 4. 수량 차감 및 비동기 처리를 위한 메시지 큐 발행
        await redis.decr(promoStockKey);
        await messageQueue.publish("promo-apply-events", {
            userId,
            promoCode,
            appliedAt: new Date()
        });
        
        return true;
    } catch (error) {
        // 에러 발생 시 롤백 로직 (필요에 따라 수량 복구)
        console.error(`프로모션 적용 실패: ${error.message}`);
        throw error;
    } finally {
        // 5. 작업 완료 후 안전하게 락 해제
        await redis.del(lockKey);
    }
}
```

이런 식으로 사용자의 직접적인 DB 쓰기 요청을 차단하고, 빠른 In-Memory 레이어(Redis)에서 1차 방어선을 구축한 뒤, 실제 영속성 저장소(DB) 반영은 메시지 큐를 통해 느슨하게 연결(Loosely Coupled)하여 처리했습니다. 덕분에 이벤트 당일, 평소의 20배가 넘는 트래픽이 몰렸음에도 단 1초의 장애 없이 깔끔하게 이벤트를 마칠 수 있었습니다. 

이 성공 경험은 저희 개발 팀원들 모두에게 엄청난 자신감을 안겨준 기념비적인 사건이 되었죠.

## 단순한 화장품 쇼핑몰이 아니다: '뷰티 테크'와 아키텍처의 확장성

다시 얼타(Ulta) 이야기로 돌아와서, 요즘의 글로벌 리테일 대기업들은 단순한 커머스 기업이 아닙니다. 이들은 고도화된 **'뷰티 테크(Beauty Tech)'** 기업으로 체질을 개선하고 있습니다. AR(증강 현실)을 이용한 가상 메이크업 테스트, AI 기반의 개인 맞춤형 피부 진단 솔루션 등이 모두 웹과 앱 내에서 API 통신을 통해 실시간으로 이루어집니다.

이런 무거운 뷰티 테크 서비스들이 동작하는 와중에 50% 할인 프로모션 트래픽까지 얹어진다면? 아키텍처가 마이크로서비스(MSA) 형태로 유연하게 쪼개져 있지 않고, 서킷 브레이커(Circuit Breaker) 같은 장애 전파 방지 대책이 없다면 도미노처럼 시스템 전체가 무너질 것입니다.

예를 들어, 가상 메이크업 시뮬레이션 API 서버가 트래픽 폭주로 마비되더라도, 핵심 비즈니스인 '결제 및 쿠폰 적용' 서비스는 아무런 영향 없이 독립적으로 살아남아야 합니다. 이것이 바로 우리가 끊임없이 시스템을 모니터링하고 가상 부하 테스트(Load Test)를 수행해야 하는 이유입니다.
